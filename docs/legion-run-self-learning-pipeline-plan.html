Executive Summary

Decision 1

Keep legion-orchestrate separate

legion-orchestrate remains a playbook/skill for decomposition, fan-out, cross-review, synthesis, and judgement-heavy orchestration. It should not be collapsed into legion-run.

Decision 2

Make legion-run the lifecycle command

legion-run is the fixed heavy-task lifecycle for work that needs proof: doctor, hints, plan, route, fan-out/apply, review, validation, evaluation, reports, share, learning, and heal planning.

Decision 3

Learning must be terminal

Self-learning cannot be just another late pipeline stage. It must be a terminal finalizer that runs on success, failure, timeout, interrupt, and uncaught exception.

Core guarantee: every official Legion-controlled terminal path produces an audit artifact, extracts learnable outcomes, promotes them into hintable memory, or records why learning could not run.

Diagnosis

The session investigation found that the specific caveman failure was not the real root problem. It was a useful symptom. Doctor did detect the defect, but the learning loop did not promote that finding into durable memory.

Observation Meaning Pipeline gap
doctor.json contained a concrete entity finding such as skill:caveman. The signal existed. The signal was not harvested as a first-class outcome.
self-learn.json recorded a coarse heavy-task failure. The failure path did some recording. Attribution was too broad and lost the underlying entity.
self-learn.json had run.skipped: true. Memory synthesis did not run on that failure path. Raw outcomes were not promoted to harness-memory.json.
legion-fanout can emit JSON with failed > 0 and still exit 0. Exit status alone is not enough. Stages need semantic validation of their output contracts.
subprocess.run() calls have no timeout. A child process can leave a run stuck at running. No terminal finalizer runs for hangs.
execute() catches only LegionRunError. Python exceptions, interrupts, and some exits bypass normal finalization. Bad-path coverage is incomplete.
Why self-learn did not trigger on its own: the old model treated self-learning as a normal late stage. If the run failed before that stage, the failure handler recorded a coarse event and skipped memory application. That made the outcome visible as a file but invisible as a future hint.

First Principles

Feedback loop

Learning is not logging

A JSON artifact is only evidence. Self-learning requires a closed loop: signal, attribution, promotion, retrieval. If any step is missing, future runs do not improve.

  1. Signal: a stage emits structured evidence.
  2. Attribution: evidence maps to an entity.
  3. Promotion: outcome becomes durable memory.
  4. Retrieval: future runs load memory as hints.
Safety

Always learn, do not always patch

Automatic durable memory is safe. Automatic source mutation is not. Legion should always record and apply memory. Source edits from self-learning should remain opt-in and scorecard-gated.

This keeps the system business-safe: failures become future guardrails without silently rewriting the toolchain.

ObserveStage artifacts, spans, stderr, validation JSON, doctor findings.
NormalizeConvert heterogeneous outputs to outcome records.
AttributeAttach each outcome to plugin, skill, command, agent, tool, or run.
PromoteRun self-learning memory synthesis.
RetrieveNext run reads hints before planning.

Self-Learn vs Self-Heal

These are separate loops and should stay separate in the product story.

Loop Question it answers Output Default safety posture
Self-learn What should future Legion runs remember before they start? outcomes.jsonl, harness-memory.json, hints, proposals. Automatic memory is safe. Source mutation stays opt-in.
Self-heal What repair should happen because this run failed or degraded? heal-plan.json and, in explicit heal mode, a candidate fix. Planning is automatic. Applying source fixes should remain explicit and gated.
Business phrasing: self-learning reduces future developer parity by turning validated misses into shared operating memory. Self-healing reduces recovery time by producing a repair path when something breaks.

Target Contract

The new contract should be explicit and testable. It should apply to legion-run and, later, any official command that starts a Legion-controlled run context.

legion.run.finalization.v1

For every terminal run:
  - write failure.json or summary.json
  - write stage-status.json
  - write artifact-manifest.json
  - write legion-observability.html
  - extract learning outcomes from every available artifact
  - append outcomes to self-learn/outcomes.jsonl
  - run legion-self-learn run --apply-memory best-effort
  - write self-learn.json with learning feedback + record + run + errors
  - write heal-plan.json best-effort
  - if learning fails, record that learning failed

What "always" means

Term Required behavior
Always attempted Every terminal path calls the finalizer. The finalizer attempts outcome extraction and memory application.
Always visible If learning cannot run, the run still writes self-learn.json and explains the failure.
Always bounded Learning finalization has a timeout and cannot hang the command forever.
Not always source-mutating Memory is automatic. Source patching remains explicit with --apply-source or a controlled auto-heal mode.

Architecture Plan

1. Consolidate terminal finalization

This pass introduced a single legion-run self-learning finalizer and artifact harvesters. A future cleanup can extract that into a shared module under legion-orchestrate/scripts/ or legion-observability/scripts/ if other commands need the same contract.

class RunFinalizer:
    def mark_stage_running(stage, artifacts): ...
    def mark_stage_passed(stage, payload): ...
    def mark_stage_failed(stage, error, payload=None): ...
    def harvest_outcomes(): ...
    def apply_memory(): ...
    def write_reports(): ...
    def finalize_success(): ...
    def finalize_failure(error): ...
    def finalize_interrupt(signal): ...
    def finalize_uncaught_exception(exc): ...

2. Make learning outcome extraction artifact-driven

Do not rely on validation/eval only. Extraction now covers the main legion-run artifacts, with additional adapters still useful for richer review and timeout evidence.

Artifact Extracted outcome Entity source
doctor.json Each fail or high-risk warn becomes an outcome. Use embedded entity, else tool:legion-doctor.
fanout.json Failed slices, blocked dependencies, apply conflicts. Slice archetype, target metadata, or command:legion-fanout.
review.json Blocking findings and reviewer rejection. Current pass validates rejecting review payloads. Rich finding extraction remains next work.
validation.json learning_feedback, failed gates, invariant discoveries. Validator-provided target, else runner entity.
eval.json learning_outcomes, failed rubric dimensions. Evaluator-provided target, else runner entity.
failure.json Stage failure summary. Runner entity.
Unhandled exception Exception type, stage, stack summary. next pass command:legion-run plus runner entity.

3. Validate stage payload semantics

Each stage needs both an exit-code check and a payload validator. Examples:

  • doctor: fail if any item has severity == "fail".
  • fanout: fail if failed > 0 or apply_conflicts > 0.
  • review: fail if verdict is reject/blocking.
  • validate: fail if exit nonzero or JSON says ok: false.
  • evaluate: fail if exit nonzero, ok: false, or score below configured threshold.

4. Add timeout and signal control

Next pass. Replace plain subprocess.run() with a stage runner that supports timeout, process groups, and kill-on-cancel. A hung fanout or model CLI must finalize as a timeout, not leave stage-status.json at running.

5. Catch all terminal paths

Next pass. legion-run should catch and finalize:

  • LegionRunError
  • subprocess.TimeoutExpired
  • KeyboardInterrupt
  • SystemExit where practical
  • generic Exception

6. Keep legion-orchestrate independent

legion-orchestrate remains its own skill/playbook. Its docs should say:

  • Use legion-run when you want the guaranteed lifecycle and self-learning finalizer.
  • Use manual legion-fanout only for debugging, experimentation, or advanced orchestration.
  • When manual orchestration is used, emit explicit outcomes with legion-self-learn record or a future legion-finalize helper.

Benchmark and Test Matrix

The current deterministic benchmark proves the happy path. The missing benchmark is a bad-path matrix that proves self-learning finalization under failure.

Case Expected finalizer behavior Must assert
Doctor failure Harvest each doctor finding as its own outcome. Hints exist for the finding entity after memory application.
Plan command fails Record runner failure and preserve plan stderr. failure.json, self-learn-run.json, and HTML exist.
Malformed plan or slices Record parser/schema failure. Outcome target is command:legion-run and runner entity.
Route failure Record failed archetype and route stderr. Memory hint references route/archetype problem.
Fanout JSON says failed but exits 0 Semantic validator fails the stage. fanout-apply marked failed and learning applied.
Fanout exits nonzero Record stage failure and fanout payload. Existing negative test upgraded to assert memory application.
Review rejects Record reviewer findings. Blocking findings visible in HTML and outcomes.
Validation fails with feedback Record both failure and validator-provided lessons. Validation feedback updates memory even when command exits 1.
Eval fails with outcomes Record eval rubric failures and learning outcomes. Eval-specific hints exist after finalization.
Report/share fails Do not lose learning because observability failed. Finalizer writes minimal fallback HTML.
Self-learn itself fails Record a meta-outcome for learning failure. self-learn.json says learning failed, no silent pass.
Timeout Kill child process, mark timeout, finalize learning. No stuck running stage.
Interrupt Finalize partial artifacts and exit nonzero. Partial HTML and learning attempt exist.

Existing and next benchmark suite

legion-bench run --suite legion-run-bad-paths --repo . --json --strict

The current test pass adds deterministic no-spend Bats coverage for the most important bad paths inside tests/legion-run.bats. A named benchmark suite can wrap those scenarios for public demo use. Each case should verify that memory was applied, not merely that self-learn.json exists.

Rollout Plan

  1. Land first-class legion-run surface. The current worktree includes the skill/docs/bin/package surface; keep it in the PR.
  2. Add finalizer module. Current pass consolidated legion-run finalization. Extract a shared module only when another command needs the same API.
  3. Add artifact outcome harvesters. Current pass covers doctor, fanout, validation, eval, and failure artifacts.
  4. Add semantic stage validators. Current pass makes stage pass/fail depend on output contracts, not exit code alone.
  5. Add timeouts and interrupt finalization. Prevent stuck runs from bypassing learning.
  6. Add bad-path benchmark suite. Fail the build if any official bad path skips memory application.
  7. Update docs. Document the exact guarantee: outcomes and durable memory are automatic; source mutation remains opt-in.
  8. Only then market it. The business story becomes defensible: Legion does not only run agents; it turns validation, review, and failure evidence into reusable operational memory.
Risk to avoid: do not claim "Legion self-improves on every run" unless the finalizer has tests for every terminal path. The accurate claim after this work is: "Legion attempts learning finalization on every official run and records why if it cannot."
